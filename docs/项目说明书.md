# ShadowLink 项目说明书

## 一、项目概述

ShadowLink 是一个本地优先的 AI Agent 工作台，面向个人知识工作、研发辅助和垂直场景智能化生产。项目采用 React Web 前端、FastAPI AI 服务、Spring Boot 网关/业务服务的三端架构，围绕 Agent 对话、RAG 知识检索、文件处理、工具调用、面试学习工作台、阅读理解工作台以及外部 Agent Run 能力进行设计。

从能力定位上看，ShadowLink 不只是一个聊天机器人，而是一个可扩展的 Agent 研发工具链原型：它把大语言模型、检索增强生成、多 Agent 编排、工具调用、MCP 协议、代码与文件处理、异步任务执行记录等能力组合起来，形成可迁移到研发管线、内容生产管线和垂直业务流程中的 AI Agent 基础设施。

结合游戏研发场景，ShadowLink 可作为 AI 驱动研发工具链的实验平台，用于探索代码审查、自动化测试辅助、策划配置生成、代码生成、资产需求拆解、知识库问答、研发文档检索、版本升级支持和玩家体验优化等方向。

## 二、项目要解决的痛点

### 1. AI 对话缺乏项目上下文

传统通用大模型只能依赖用户一次性输入，难以稳定理解项目文档、代码结构、业务规则和历史资料。对于研发团队而言，需求文档、技术方案、配置表、代码仓库和运行日志分散在不同位置，导致 AI 难以直接参与真实生产流程。

ShadowLink 通过 RAG 知识库、文件解析、知识检索工具和 Agent 上下文注入，将项目资料转化为可检索、可引用、可推理的上下文，使 AI 能围绕具体项目进行回答、分析和执行。

### 2. 研发流程中的重复劳动成本高

研发过程中存在大量重复性任务，例如阅读长文档、总结需求、生成配置、检查代码、编写测试、定位问题、整理面试材料和复盘历史方案。这类任务对人力消耗大，但又需要结合上下文和工具链才能完成。

ShadowLink 使用 Agent 编排与工具调用，把“理解任务—拆解步骤—检索资料—调用工具—输出结果”的过程产品化，降低重复劳动成本，为后续接入代码审查、自动化测试、配置生成和代码生成等研发工具提供基础。

### 3. 工具链割裂，AI 难以进入真实工作流

许多 AI 应用停留在问答层，无法调用本地文件、知识库、代码工具或外部系统，也难以沉淀执行过程。ShadowLink 将工具统一抽象为 Agent 可调用能力，并通过 MCP 协议预留跨工具互操作空间，使 Agent 可以从“回答问题”进一步走向“执行任务”。

### 4. 数据隐私与本地化部署诉求强

研发资料、简历材料、项目文档、游戏策划案和代码资产通常具有较高敏感性，完全依赖云端 AI 存在隐私与合规顾虑。ShadowLink 采用本地优先设计，支持本地服务启动、本地存储和本地模型/外部模型灵活接入，便于在个人或团队内网环境中演进。

### 5. 游戏研发需要跨职能协作型 AI 工具

游戏研发涉及策划、美术、程序、测试、运营等多角色协作，生产链路长、资产类型复杂、迭代频繁。ShadowLink 的 Agent 工具链思想可映射到游戏工业化管线：让 AI Agent 理解设计文档、读取配置表、辅助生成代码、审查改动、解释资源规范，并在多角色协作中承担“上下文桥梁”和“流程自动化助手”的角色。

## 三、核心功能

### 1. Agent 智能对话

项目提供 Agent 对话能力，支持前端流式交互和后端 Agent 编排。Agent 不仅生成自然语言回答，还可以结合当前模式、系统提示词、可用工具、RAG 上下文和执行步骤进行任务处理。

适用场景包括：

- 项目知识问答与技术方案解释。
- 需求拆解、学习规划和任务规划。
- 研发问题分析与解决方案建议。
- 面向游戏研发的策划案分析、技术方案评估和管线优化建议。

### 2. RAG 知识检索与知识库问答

ShadowLink 内置 RAG Pipeline，支持文档解析、向量检索、结果融合、重排序、质量门控、上下文组装和答案生成。项目依赖 FAISS、sentence-transformers、LangChain/LangGraph 等组件，能够把本地资料转化为 Agent 可用的知识上下文。

RAG 能力可用于：

- 检索项目文档、技术方案和学习资料。
- 为 Agent 回答提供依据，减少幻觉。
- 支持游戏项目中的策划文档、配置说明、接口文档和美术规范检索。
- 为代码审查、测试建议和版本升级说明提供历史上下文。

### 3. 文件处理与多格式资料接入

AI 服务支持 PDF、DOCX、XLSX、PPTX、Markdown、代码文件和图片等多类型资料处理，为知识库构建和 Agent 分析提供输入基础。

这一能力对于游戏研发尤其重要，因为真实生产资料往往同时包含策划文档、表格配置、美术规范、接口说明、版本日志和代码片段。多格式解析让 Agent 可以覆盖更完整的研发上下文。

### 4. 工具调用系统

项目将本地能力封装为可被 Agent 调用的工具，例如知识库搜索、文件操作、代码执行等。所有内置工具继承统一的基础工具接口，并预留 MCP ToolInfo 转换能力，便于未来接入更多外部工具。

工具调用能力使 Agent 可以完成更接近研发生产的任务：

- 检索资料后再回答。
- 读取和分析本地文件。
- 执行受控代码或脚本进行验证。
- 对接外部研发系统、测试系统、资源管理系统或游戏引擎工具链。

### 5. 面试学习工作台

项目提供面试学习工作台，支持 Job Space、Space Profile、简历/JD/目标公司/目标岗位管理、面试回答 Review 草稿、外部 Agent Run 记录等能力。

该模块展示了 ShadowLink 的垂直场景落地方法：围绕一个明确任务空间组织资料、上下文、交互记录和 Agent 执行过程。类似设计可以迁移到游戏研发中的“版本 Space”“功能 Space”“关卡 Space”“资产 Space”或“Bug Space”。

### 6. 阅读理解工作台

阅读工作台支持文章切分、句子解释、阅读进度保存、难句标记和 Space 隔离，适合用于技术文档、论文、岗位 JD、游戏设计资料和英文资料阅读。

这一能力体现了 Agent 在知识吸收环节的应用：将长资料拆解为可理解、可追踪、可复盘的学习单元，帮助研发人员快速掌握新技术、新引擎特性或新工具流程。

### 7. 外部 Agent Run 与 Codex CLI 专家模式预留

项目为 External Agent Run 和 Codex CLI 专家模式预留了异步运行记录骨架，可记录外部 Agent 的运行请求、任务 Prompt、状态和结果。

这一设计使 ShadowLink 具备扩展为“多 Agent 研发平台”的潜力：不同 Agent 可以分别负责代码分析、测试生成、配置检查、文档总结、资产需求拆解等任务，并把结果回流到统一工作台中。

## 四、Agent 技术设计

### 1. ReAct Agent

项目包含 ReAct Agent 思路，即让模型在推理和行动之间循环：先理解问题并形成思考，再选择工具执行，最后根据工具结果继续推理或给出答案。

ReAct 适用于需要动态调用工具的任务，例如：

- 先检索知识库，再回答项目问题。
- 先读取文件，再总结文档。
- 先执行代码片段，再判断问题原因。
- 在游戏研发中先检查配置或代码，再给出修复建议。

### 2. Plan-and-Execute Agent

项目包含 Plan-and-Execute Agent 模式，即先生成任务计划，再按步骤执行，并支持动态调整。这类模式适合复杂、多步骤、需要过程可视化的任务。

在研发场景中，它可以用于：

- 拆解功能开发任务。
- 规划代码审查步骤。
- 生成自动化测试方案。
- 分析游戏版本升级流程。
- 将策划需求转化为程序、美术和测试的执行清单。

### 3. Multi-Agent 与专家协作

项目预留 MultiAgent 架构，支持 Supervisor 调度多个专家 Agent。不同 Agent 可以面向不同任务角色，例如知识检索 Agent、代码分析 Agent、测试 Agent、文档 Agent、策划配置 Agent 或美术资产需求 Agent。

映射到游戏研发管线，可以形成如下协作模式：

- 策划 Agent：分析玩法需求，生成配置草案和规则说明。
- 程序 Agent：生成代码方案、接口草案和改动建议。
- 测试 Agent：生成测试用例、边界条件和自动化测试建议。
- 美术 Agent：拆解资产需求、整理风格约束和资源清单。
- Supervisor Agent：汇总多方结果，形成可执行研发计划。

### 4. TaskComplexityRouter 任务复杂度路由

项目的 Agent Engine 设计中包含任务复杂度路由思想，可根据任务复杂度选择更合适的执行策略。简单问题可以直接回答，复杂任务交给 ReAct、Plan-and-Execute 或 MultiAgent 流程。

该设计体现了对 Agent 能力边界的理解：并非所有任务都需要复杂编排，系统应在成本、延迟、可靠性和可解释性之间做权衡。

### 5. Agent Memory 记忆系统

项目包含短期记忆、长期记忆、语义记忆和情节记忆等模块结构，用于支持 Agent 在会话和任务维度上沉淀上下文。

记忆能力可以帮助 Agent：

- 保留用户偏好和项目背景。
- 复用历史任务经验。
- 追踪学习进度与问题记录。
- 在游戏研发中记住项目规范、团队约定、配置格式和常见错误模式。

### 6. MCP 协议与工具互操作

ShadowLink 引入 MCP 相关依赖和工具信息转换能力，目标是让 Agent 不局限于内置函数调用，而是可以与标准化外部工具生态互联。

对于游戏研发工具链，这意味着后续可以把引擎编辑器、构建系统、测试平台、资源管理系统、配置表工具和缺陷跟踪系统封装为可调用工具，由 Agent 统一调度。

### 7. 多模型 Provider 接入

项目支持多模型 Provider 思路，包括 OpenAI、DeepSeek、Ollama、Anthropic 等模型来源。该设计便于在不同任务中选择不同模型：云端强模型用于复杂推理，本地模型用于隐私敏感或低成本任务，专用模型用于嵌入、重排序或代码处理。

## 五、面向游戏研发管线的能力映射

虽然 ShadowLink 当前是通用 AI Agent 工作台，但其技术架构与游戏研发 Agent 工具链高度契合，可扩展到以下方向。

### 1. 代码审查 Agent

基于代码文件解析、RAG 项目规范检索、ReAct 工具调用和外部 Agent Run，可以构建代码审查 Agent：自动阅读改动、对照项目规范、识别潜在风险、生成 Review 建议，并把结果沉淀为审查记录。

### 2. 自动化测试与优化 Agent

基于 Plan-and-Execute 和代码执行工具，可以让 Agent 根据需求或 Bug 描述生成测试计划、边界用例、回归测试建议和性能优化方向。未来接入 CI、单元测试框架或游戏自动化测试工具后，可进一步实现测试执行与结果解释。

### 3. 策划配置生成 Agent

基于 RAG 检索历史配置规则、文件处理读取表格、Agent 生成结构化内容，可以辅助策划生成技能配置、关卡参数、任务表、数值草案和版本改动说明，并由程序或测试 Agent 进一步校验格式与逻辑一致性。

### 4. 程序代码生成 Agent

结合项目上下文、代码规范、接口文档和外部 AI 编程工具，ShadowLink 可扩展为代码生成与代码修改入口，帮助程序完成样板代码、接口适配、工具脚本和原型功能开发。

### 5. 美术资产生成与需求拆解 Agent

通过多模态模型或外部资产生成工具接入，Agent 可以根据策划需求生成美术需求清单、Prompt 草案、风格约束、资源命名规范和验收标准。ShadowLink 当前的工具抽象和外部 Agent Run 机制为此类工具接入提供了基础。

### 6. 引擎与工具流定制

ShadowLink 的 MCP/工具调用设计可以扩展到 Unity、Unreal 或自研引擎工具链。通过把引擎命令、资源导入、配置校验、构建打包和日志分析封装为工具，Agent 可以协助团队重构工业化管线，形成 AI 驱动的敏捷研发模式。

## 六、技术架构

### 1. 前端层：shadowlink-web

前端采用 React、Vite、TypeScript、React Router、TanStack Query、Zustand、Axios、Framer Motion、Tailwind CSS 等技术，提供聊天页、知识库页、设置页、面试学习页和阅读页。

前端负责：

- Agent 对话与流式结果展示。
- 工具调用、计划步骤和执行状态可视化。
- 知识库、阅读、面试等垂直工作台交互。
- 与 Python AI 服务和 Java 网关通信。

### 2. AI 服务层：shadowlink-ai

AI 服务采用 FastAPI、Pydantic、LangChain、LangGraph、FAISS、sentence-transformers、MCP、gRPC、structlog 等技术，承担核心智能能力。

AI 服务负责：

- Agent 编排与策略路由。
- RAG 检索增强生成。
- LLM Provider 管理。
- 文件解析与知识处理。
- 工具注册、执行和 MCP 兼容。
- 面试学习、阅读工作台、外部 Agent Run 等业务接口。

### 3. Java 网关/业务层：shadowlink-server

Java 服务采用 Spring Boot 多模块结构，包含 AI Bridge、Auth、Business、Common、Starter 等模块，定位为网关、认证、业务扩展和企业化接入层。

该层可用于：

- 对外提供统一业务 API。
- 对接 Python AI 服务。
- 承载团队级权限、认证和业务数据。
- 为未来多人协作、项目空间和企业级部署提供基础。

### 4. 桌面化预留：shadowlink-electron

项目保留 Electron 壳，便于未来将 Web 工作台打包为桌面应用，适合本地优先、研发工具链和内网环境使用。

## 七、工程能力与可迭代性

ShadowLink 体现了完整的工程化思路：

- 前后端分层清晰，便于独立开发和部署。
- Python AI 服务通过 pyproject 管理依赖，预留测试、Lint、类型检查和覆盖率配置。
- 前端提供构建、类型检查、Lint 等工程脚本。
- Java 服务采用多模块架构，便于扩展业务和网关能力。
- 工具、Agent、RAG、LLM Provider、业务场景模块化组织，便于持续迭代。

这些能力与岗位中“借助 AI 编程工具提升开发效率，完成原型开发、系统实现和持续迭代”的要求相匹配，也为游戏研发工具链原型开发提供了可复用基础。

## 八、与岗位 JD 的对应关系

### 1. 前沿 AI Agent 技术研究与落地

项目覆盖 ReAct、Plan-and-Execute、MultiAgent、任务复杂度路由、Agent Memory、RAG、Tool Calling、MCP、多模型 Provider 等 Agent 关键技术，并将其落地为可运行的 AI 工作台，而不是停留在概念层。

### 2. 基于 AI Agent 的研发工具链设计

项目的工具调用、RAG、文件处理、外部 Agent Run 和多工作台设计，可直接扩展到代码审查、自动化测试、代码生成、配置生成、文档分析和资产需求拆解等研发工具链场景。

### 3. 与研发团队协作并定制工具流

项目采用本地优先和模块化架构，便于接入团队内部服务、引擎工具、测试平台、资源系统和知识库。通过 MCP 和工具抽象，可以把团队已有工具封装为 Agent 可调用能力，服务策划、美术、程序和测试多角色协作。

### 4. 大语言模型与 Agent 能力边界理解

项目通过 RAG 减少幻觉，通过复杂度路由控制成本和执行策略，通过工具调用弥补模型无法直接操作环境的问题，通过本地优先和多 Provider 接入平衡隐私、成本和能力。

### 5. 编程与工程实现能力

项目横跨 Python、TypeScript、Java 三类技术栈，包含 AI 服务、Web 前端、Java 网关、文件处理、向量检索、gRPC/MCP、工程脚本和多端启动流程，体现从原型到系统化实现的工程能力。

### 6. 游戏开发流程理解与跨领域视野

项目虽然当前不是专用游戏引擎插件，但其能力可映射到游戏研发全流程：策划案理解、配置表生成、程序代码辅助、测试用例生成、美术资产需求拆解、构建与日志分析、版本升级和玩家反馈总结。

## 九、典型使用场景

### 1. 个人 AI 研发助手

开发者可以把技术文档、项目说明、代码片段和学习资料导入 ShadowLink，通过 Agent 对话、RAG 检索和阅读工作台提升学习与研发效率。

### 2. 面试与岗位能力准备

用户可以维护不同岗位的 Job Space，保存简历、JD、目标公司和学习记录，并让 Agent 辅助拆解岗位要求、复盘回答和补齐技术短板。

### 3. 团队知识库与研发问答

团队可以将项目文档、接口规范、配置说明和历史方案接入知识库，让新成员或跨职能成员通过自然语言快速获取上下文。

### 4. 游戏研发 Agent 工具链原型

研发团队可以基于 ShadowLink 的 Agent、RAG、工具调用和外部 Run 架构，逐步接入代码仓库、CI、配置表、引擎命令和资源管理工具，构建面向游戏生产的 AI 工具链。

7. 探索多模态资产生成，将文本需求、视觉参考和资源规范连接到美术资产生产流程。

## 十一、总结

ShadowLink 是一个以 AI Agent 为核心的本地优先工作台项目。它通过 Agent 编排、RAG、工具调用、MCP、多模型接入和多端工程架构，把大语言模型从单纯问答扩展为可参与研发流程的智能工具链。

对于游戏研发方向，ShadowLink 展示了将 AI Agent 应用于工业化管线的可行路径：让 Agent 理解项目上下文，调用研发工具，协同策划、美术、程序和测试角色，逐步落地到代码审查、自动化测试、配置生成、代码生成、资产需求拆解和玩家体验优化等环节。

该项目既体现了对前沿 Agent 技术的理解，也体现了将 AI 能力工程化、产品化并服务复杂研发流程的实践能力。
