# ShadowLink 分层架构设计

> **Version**: 1.0\
> **Date**: 2026-04-16\
> **Status**: Active\
> **Goal**: 用分层演进方式推进整个项目，保证每一层都可开发、可演示、可向外展示

***

## 一、为什么要做分层架构

ShadowLink 不是一个单点功能，而是一个完整的 AI 工作台产品。

如果一开始就按“终局架构”同时建设所有模块，结果通常会是：

- 技术点很多，但没有一条完整链路可跑
- 模块都在开发，但产品没有可展示版本
- 每层都想一次做完，最终每层都半成品

所以这个项目更适合采用 **分层演进架构**：

> 先把最上层、最可见、最能体现产品价值的链路做成 demo，\
> 再逐层把中层、下层、基础层替换成更完整的正式实现。

这不是“先做假东西”，而是：

- 先做可运行的产品外壳
- 再逐步把内部替换成更强的能力

***

## 二、这份文档的核心思想

这份文档不是传统“技术分层图”。

它采用的是 **产品展示层 + 能力编排层 + 领域能力层 + 基础支撑层 + 高级增强层** 的分层方式。

每一层都回答四个问题：

1. 这一层的目标是什么
2. 这一层用户能看到什么
3. 这一层技术上先做什么
4. 这一层哪些内容可以先占位

这样做的好处是：

- 对开发有路线
- 对产品有演示节奏
- 对汇报有故事线
- 对后续重构有边界

***

## 三、总体分层模型

ShadowLink 建议按五层推进：

```text
L4 高级增强层
    多 Agent / 高级 RAG / 可观测 / 生态扩展

L3 基础支撑层
    正式数据模型 / RAG 索引 / 记忆 / 工具注册 / 安全与稳定性

L2 领域能力层
    Agent / WorkMode / Knowledge / Session 的正式能力实现

L1 能力编排层
    Java-Python-Frontend 的主调用链和编排逻辑

L0 产品展示层
    可运行、可交互、可演示的完整产品外壳
```

这个模型的关键点是：

- 不是先从最底层开始做
- 而是先让 `L0 + L1` 跑起来
- 再把 `L2 + L3` 一层层补实
- 最后再上 `L4`

***

## 四、分层原则

### 4.1 上层先成立，再要求下层完美

只要上层产品体验已经能跑通，下层先用最小实现也可以。

例如：

- Agent 可以先用轻量规则路由
- RAG 可以先用基础向量检索
- Memory 可以先只做短期上下文
- Tool 可以先只保留 2-3 个核心工具

### 4.2 每层都必须有“展示点”

每一层都应该有可以拿出去演示的内容，而不是只有代码结构优化。

这意味着设计每层时，必须同步考虑：

- 用户可见价值
- 演示脚本
- 页面效果
- 输出结果可解释性

### 4.3 每层都必须可替换

上层不应强依赖下层的复杂内部实现。

例如：

- 前端只关心事件流，不关心 Agent 内部图结构
- Java 只关心 AI 服务接口，不关心 Python 内部策略实现
- Agent 只关心 Knowledge 接口，不关心底层索引类型

只有这样，后面才能平滑替换实现。

### 4.4 从“可运行”到“可用”再到“可扩展”

项目推进顺序应该是：

1. 可运行
2. 可演示
3. 可用
4. 可扩展

不要直接跳到第 4 步。

***

## 五、L0：产品展示层

### 5.1 这一层的目标

L0 的目标不是“技术正确”，而是：

> 让 ShadowLink 看起来已经是一个能工作的产品。

这一层是最先应该打通的，因为它决定：

- 你能不能做 demo
- 你能不能给别人看
- 你能不能证明产品方向成立

### 5.2 这一层用户能看到什么

L0 至少要有以下产品表象：

- 登录后进入主工作台
- 能切换模式
- 能发起一次对话
- 能看到流式回复
- 能看到引用来源或工具状态
- 能进入知识库页面
- 能看到会话列表

换句话说，L0 是“产品壳”。

### 5.3 L0 的核心价值

L0 不负责让每个能力很强，它负责让用户第一眼感受到：

- 这不是普通聊天框
- 这是一个模式化的 AI 工作台
- 它有工作流、有状态、有上下文、有知识库

### 5.4 L0 技术上先做什么

前端先做：

- 主工作台布局
- 模式切换入口
- 聊天面板
- SSE 流式渲染
- 简单知识库管理页
- 简单设置页

后端先保证：

- 登录接口可用
- 对话接口可调用
- 会话列表可取回
- 模式列表可取回
- 知识源列表可取回

### 5.5 L0 可以先占位什么

- 模式编辑器先做简化版
- 引用来源先做简单列表
- Agent 过程面板先做轻量展示
- 知识库 ingest 进度先做状态文本
- UI 动画和氛围感效果先做基础版本

### 5.6 L0 的演示价值

L0 可以支撑以下展示话术：

- “这是一个基于工作模式的 AI 工作台”
- “用户切换模式后，系统行为会变化”
- “系统不是一次性回答，而是有流式执行过程”
- “可以接入知识库，而不是纯聊天机器人”

***

## 六、L1：能力编排层

### 6.1 这一层的目标

L1 解决的是：

> 项目各个服务和页面之间，怎么串成一条完整主链路。

这一层是整个项目的“运行总线”。

### 6.2 L1 包含哪些内容

- 前端 -> Java 的 REST / SSE 链路
- Java -> Python 的 AI 调用链路
- 请求上下文传递
- session\_id / mode\_id 透传
- 流式事件转发
- 错误处理和回退

### 6.3 为什么 L1 很关键

很多项目失败，不是因为模型不好，而是因为调用链根本没打通。

L1 打通之后，你就已经拥有：

- 一条可运行的端到端路径
- 一个真实可交互的 demo
- 一个后续所有能力都能挂载的骨架

### 6.4 L1 技术设计

首版建议保持简单：

- 前端 -> Java：REST + SSE
- Java -> Python：HTTP
- 不强依赖 gRPC
- 不强依赖 WebSocket

这样能最小化调试成本。

### 6.5 L1 可先占位什么

- 某些事件先合并为简单类型
- 某些状态先不做精细错误码
- Java 先做轻代理，不做复杂治理
- Python 先暴露最小 Agent / RAG 接口

### 6.6 L1 的演示价值

L1 能支撑以下展示点：

- 一次消息从前端传到 AI 服务并流式返回
- 当前会话和当前模式被完整识别
- 系统不是本地假交互，而是真正服务协同

***

## 七、L2：领域能力层

### 7.1 这一层的目标

L2 才是真正定义 ShadowLink“产品差异化能力”的地方。

这一层主要解决四个核心领域对象：

- WorkMode
- Agent
- Knowledge
- Session

### 7.2 L2 为什么是核心层

L0 让产品能看，L1 让系统能跑，而 L2 决定产品有没有灵魂。

ShadowLink 的灵魂不在“接了 LLM”，而在：

- 模式是否真的影响系统行为
- Agent 是否真的会选择合适策略
- Knowledge 是否真的能提升回答质量
- Session 是否真的保存工作过程

### 7.3 L2 的四个核心模块

#### WorkMode

作用：

- 承载场景上下文
- 约束工具范围
- 约束知识范围
- 影响 prompt 和策略偏好

首版最小目标：

- 能创建
- 能切换
- 能影响后续对话

#### Agent

作用：

- 识别任务复杂度
- 选择执行策略
- 驱动 direct / react / plan\_execute

首版最小目标：

- 自动路由可用
- ReAct 可用
- Plan-and-Execute 可跑

#### Knowledge

作用：

- 接管文档摄入、索引、检索
- 为 Agent 提供知识增强上下文

首版最小目标：

- 文件可 ingest
- 索引可用
- 检索可命中
- 回答可返回引用来源

#### Session

作用：

- 保存工作过程
- 支持回到历史对话
- 给 Agent 提供基础上下文

首版最小目标：

- 会话能创建
- 消息能保存
- 页面能回看

### 7.4 L2 可以先用什么最小实现

- WorkMode 先只保留少数字段
- Agent 先只保留三种策略
- Knowledge 先只支持本地文件 / 本地目录
- Session 先只保存消息和基本元信息

### 7.5 L2 的演示价值

L2 是最适合展示产品差异化的一层：

- “切换到论文模式后，回答风格和知识源会变化”
- “复杂任务会自动走规划型策略”
- “问答结果能引用你自己的知识库内容”
- “会话不是临时的，而是可持续工作空间”

## 八、L3：基础支撑层

### 8.1 这一层的目标

L3 负责把 L2 的领域能力变成可长期维护的系统能力。

这一层用户不会直接感知，但它决定：

- 系统是否稳定
- 数据是否一致
- 能力是否可持续迭代

### 8.2 L3 包含哪些内容

- 正式数据模型和表结构
- 知识源状态管理
- FAISS 索引与存储目录规范
- Tool Registry
- 短期记忆 / 长期记忆的注入机制
- 配置管理
- 日志体系
- 基础权限控制

### 8.3 为什么 L3 不该一开始就做太重

如果一开始就做完整基础层，容易出现：

- 表设计很完整，但产品功能还没成立
- 日志很多，但没有真实用户路径
- 工具系统很完整，但还没明确哪些工具值得保留

所以 L3 应该跟着 L2 往下补，而不是在 L0/L1 之前先行铺满。

### 8.4 L3 首版建议实现

- MySQL 中的用户、模式、会话、消息、知识源表
- 本地文件存储和索引目录
- Tool 注册和白名单控制
- Memory context 的统一注入格式
- 请求级日志和错误日志

### 8.5 L3 可以后补的内容

- 更复杂的配置中心
- Redis 级缓存体系
- 更完整的长期记忆策略
- 更精细的审计结构

### 8.6 L3 的演示价值

L3 的展示性不在页面，而在可信度：

- “不是 demo 数据，而是有真实会话存储”
- “知识库不是临时拼接，而是有独立索引状态”
- “工具不是乱调用，而是受控注册”
- “系统不是黑箱，有基本日志和错误追踪”

***

## 九、L4：高级增强层

### 9.1 这一层的目标

L4 不是首版必须层，而是用来做：

- 技术亮点
- 体验增强
- 架构扩展性
- 对外讲故事的上限

### 9.2 L4 可包含的内容

- MultiAgent
- Supervisor / Swarm
- 高级 RAG
- 更复杂的 Query Rewrite
- CRAG / Self-RAG
- 高级 Memory
- 全链路可观测
- 插件生态 / MCP 扩展
- Electron 桌面能力

### 9.3 为什么 L4 要最后做

因为这些能力虽然“好讲”，但很容易：

- 增大系统复杂度
- 稀释核心主链路
- 让首版陷入 endless building

所以 L4 的前提是：

- L0 产品已可展示
- L1 链路已稳定
- L2 差异化能力已成立
- L3 基础支撑已够用

### 9.4 L4 的展示价值

L4 更适合在后期做：

- 技术深度展示
- 项目亮点展示
- 面试 / 路演时的高级能力说明

但它不该定义项目起点。

***

## 十、推荐的推进顺序

### 10.1 第一阶段：先做 L0 + L1

目标：

- 产品壳出来
- 主链路跑通
- 能做第一版 demo

重点完成：

- 工作台页面
- SSE 流式输出
- Java -> Python 编排链路
- 基础登录
- 基础会话

### 10.2 第二阶段：补齐 L2

目标：

- 做出真正的产品差异化

重点完成：

- WorkMode 真正生效
- Agent 自动路由
- 基础 RAG
- Session 完整可用

### 10.3 第三阶段：补强 L3

目标：

- 系统从“可演示”进入“可持续开发”

重点完成：

- 表结构和状态管理稳定
- Tool Registry 完整
- memory 注入格式统一
- 日志和错误处理完善

### 10.4 第四阶段：选择性进入 L4

目标：

- 增加高级亮点

重点完成：

- MultiAgent 或高级 RAG 中优先级最高的一项
- 更强的可视化
- 更好的扩展能力

- 展示 SSE 事件持续返回

- “系统链路已经真实打通”

## 十二、这套分层架构对当前项目的具体意义

结合 ShadowLink 当前现状，最合理的结论是：

- 现在最应该优先做的是 `L0 + L1`
- 当前最应该尽快成立的是 `L2`
- `L3` 不要缺失，但不要过度建设
- `L4` 现在可以保留规划，不应该抢主线

换句话说：

> 当前阶段最重要的不是“做更多高级功能”，\
> 而是让 ShadowLink 尽快拥有一个可以反复展示、反复迭代的产品主链路。

***

## 十三、最终结论

ShadowLink 的分层架构，不应该按“技术组件完整度”来推进，而应该按“产品成形度”来推进。

最合理的节奏是：

1. 先做出能展示的产品壳
2. 再打通真实调用链
3. 再让核心领域能力真正生效
4. 再补齐基础支撑
5. 最后再上高级亮点

因此，整个项目的分层推进顺序应该是：

```text
先 L0/L1，后 L2/L3，最后 L4
```

这条路线既能保证产品持续可展示，也能保证架构持续可落地。
