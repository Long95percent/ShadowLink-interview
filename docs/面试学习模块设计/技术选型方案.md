# DTALS 技术选型方案

> 基于 `理念.txt` 的设计目标：构建一个轻量、本地优先、基于 HTTP 的 Dual-Track Adaptive Learning System，支持岗位面试 Agent 与阅读理解工作台两个解耦赛道。

## 1. 选型结论

### 推荐技术栈

| 层级 | 推荐选型 | 选择理由 |
| --- | --- | --- |
| 前端 | React + Vite + TypeScript | 本地启动快，工程轻量，适合做玻璃拟态、对比审阅、Space 切换等交互 |
| UI | Tailwind CSS + shadcn/ui | 组件轻量、可高度定制，适合实现 macOS Spotlight 式输入框与磨砂浮层 |
| 后端 | FastAPI + Uvicorn | Python 原生适配 LangChain / LangGraph，HTTP API、SSE、文件上传实现简单 |
| 通信 | HTTP REST + 短轮询 + SSE | 避免 WebSocket 的复杂握手、连接状态维护和协议层复杂度 |
| 主数据库 | SQLite | 单机本地运行零运维，天然轻量，适合个人学习系统与本地隐私数据 |
| ORM / 迁移 | SQLAlchemy + Alembic | 成熟稳定，便于管理 Resume、JD、Session、Message、Artifact 等结构化数据 |
| 向量库 | Chroma | 本地持久化简单，适合简历、JD、代码片段、文章片段的轻量 RAG 检索 |
| Agent 框架 | LangChain + LangGraph | LangChain 负责工具、模型、RAG 编排；LangGraph 负责多步骤、有状态 Agent 工作流 |
| 本地代码读取 | Serena MCP + MCP Filesystem Server + ripgrep | Serena 提供语义级代码检索，Filesystem 做受控文件读取，ripgrep 做快速文本搜索 |
| 本地代码专家模式 | Codex CLI 外部 Agent 适配器 | 面向复杂项目、宽技术栈、跨文件架构分析；默认只读运行，通过 SSE 回传过程事件 |
| 网页查阅 | 预留 Web Research Provider | 当前不购买外部搜索 API，先保留 OpenAI / Claude / Gemini / DeepSeek 等模型联网搜索适配层 |
| 数据库工具 | SQLAlchemy / Alembic / LangChain SQLDatabaseToolkit | 写库走 ORM 与事务；Agent 查询数据库走受控 Tool，避免直接拼 SQL |

## 2. 架构原则

### 2.1 本地优先与轻量运行

- 默认以本地单机应用运行：`frontend`、`backend`、`sqlite.db`、`chroma` 均在本机。
- 后端只暴露 HTTP 接口，不引入消息队列、分布式调度、WebSocket 网关等重组件。
- 大模型可接云端 API，也可预留本地模型适配层；业务数据、简历、JD、本地代码默认不上传到非必要第三方。

### 2.2 HTTP 优先，不使用复杂协议

- 前端与后端统一基于 HTTP：普通操作用 REST，任务状态用短轮询，生成过程用 SSE。
- 不推荐 WebSocket：本项目没有强实时多人协同需求，WebSocket 会增加连接保活、断线重连、代理兼容和状态同步成本。
- SSE 单向流足够满足 Agent 输出、批改进度、检索溯源逐步返回等场景。

### 2.3 多 Session 与多岗位隔离

- `Job Space` 是最高级业务隔离单元，不同岗位必须拥有独立 Resume、JD、材料、向量集合、Agent Session。
- `Session` 是一次学习 / 面试 / 阅读任务的运行上下文，必须绑定到具体 `space_id`。
- 所有消息、生成结果、检索记录、工具调用日志都必须带 `space_id` 与 `session_id`，避免 AI 工程师、PM、日语学习等上下文串线。

## 3. 前后端通信方案

### 3.1 REST API

用于确定性业务操作：

- 创建 / 切换 Job Space。
- 上传 Resume、JD、文章、代码库路径配置。
- 创建 Agent 任务。
- 保存用户采纳 / 舍弃导师建议的审阅决策。
- 查询历史 Session、消息、产物、溯源资料。

示例接口：

```text
POST /api/spaces
POST /api/spaces/{space_id}/resume
POST /api/spaces/{space_id}/jd
POST /api/spaces/{space_id}/sessions
POST /api/sessions/{session_id}/answers
POST /api/revisions/{revision_id}/accept
POST /api/revisions/{revision_id}/reject
```

### 3.2 短轮询

前后端要求采用短轮询方案，建议用于“状态类”数据，而不是 token 流：

- 任务状态：`queued`、`running`、`waiting_user_review`、`completed`、`failed`。
- 后台索引状态：简历解析、JD 解析、代码库索引、文章切片入库。
- Space 左侧列表、Session 历史、未读审阅提醒。

推荐策略：

```text
GET /api/tasks/{task_id}/status
GET /api/sessions/{session_id}/state
GET /api/spaces/{space_id}/indexing-status
```

- 前端每 1.5 到 3 秒轮询一次。
- 页面不可见时降频到 10 到 15 秒。
- 任务完成或失败后停止轮询。
- 返回 `updated_at` / `version`，前端只在版本变化时刷新局部状态。

### 3.3 SSE

SSE 用于 Agent 生成流与过程事件：

```text
GET /api/sessions/{session_id}/events
```

事件类型建议：

```text
event: token          # 模型增量输出
event: tool_start     # 工具开始执行
event: tool_result    # 工具执行结果摘要
event: source         # 网页 / 代码 / 文档溯源
event: review         # 导师批改建议生成完成
event: done           # 本轮结束
event: error          # 错误信息
```

设计边界：

- SSE 只做服务端到客户端的单向推送。
- 用户输入、采纳、舍弃、重新生成等动作仍走 REST。
- 断线后前端使用 `Last-Event-ID` 或 `cursor` 继续拉取，后端从 `agent_events` 表补发缺失事件。

## 4. 数据库选型与数据模型

### 4.1 为什么主库选 SQLite

本项目强调本地、轻量、隐私、个人学习，SQLite 更合适：

- 无需单独安装数据库服务。
- 单文件备份与迁移方便。
- 对 Resume、JD、Session、Message、Review 等结构化数据足够。
- 与 SQLAlchemy、Alembic、Chroma 的本地运行模式匹配。

后续如果需要多用户部署，可平滑迁移到 PostgreSQL；ORM 层保持 SQLAlchemy，降低迁移成本。

### 4.3 向量库隔离

Chroma collection 建议按 Space 隔离：

```text
space_{space_id}_resume
space_{space_id}_jd
space_{space_id}_code
space_{space_id}_articles
```

这样可以确保：

- AI 工程师岗位不会检索到 PM 岗位 JD。
- 日语阅读不会混入技术面试材料。
- 删除某个 Space 时可以直接删除对应 collection。

## 5. Agent 方案

### 5.1 Agent 框架

推荐组合：

- LangChain：模型封装、Prompt、Retriever、Tool、结构化输出。
- LangGraph：多步骤 Agent 状态机、Human-in-the-Loop、可恢复执行、多 Session 状态隔离。

技术面试 Agent 可建成如下图式流程：

```text
读取 Space Profile
  -> 检索 Resume / JD / 本地代码 / 网页资料
  -> 生成追问或批改
  -> 输出 [原回答 vs 修改建议]
  -> 等待用户采纳 / 舍弃 / 重新生成
  -> 写入 reviews 与 messages
```

阅读理解 Agent 流程：

```text
读取文章
  -> 用户点击句子
  -> 语法拆解
  -> 职场 / 考试语境解释
  -> 记录阅读进度与难点
```

### 5.2 Human-in-the-Loop

审阅权力归还是核心产品理念，技术上必须显式建模：

- Agent 只生成 `suggested_answer`，不得自动覆盖 `original_answer`。
- 用户点击“采纳并更新”后，才写入最终答案或学习笔记。
- 用户点击“保留原样”后，保留 critique，但不覆盖原文。
- 用户通过 Spotlight 式输入框追加约束后，创建新的 `agent_task` 重新生成局部建议。

## 6. 工具选型

### 6.1 数据库读写工具

数据库写入属于高风险操作，不建议让 LLM 直接自由拼接 SQL。推荐分层：

1. 业务写入：后端服务使用 SQLAlchemy ORM + Pydantic Schema，显式函数完成写库。
2. 数据库迁移：Alembic 管理 schema 版本。
3. Agent 查询：LangChain `SQLDatabaseToolkit` 只开放白名单查询或只读分析能力。
4. MCP 数据库工具：仅在开发调试或受控管理后台使用，不直接暴露给普通 Agent。

GitHub 地址：

- SQLAlchemy: https://github.com/sqlalchemy/sqlalchemy
- Alembic: https://github.com/sqlalchemy/alembic
- LangChain: https://github.com/langchain-ai/langchain
- LangGraph: https://github.com/langchain-ai/langgraph
- MCP Database Server: https://github.com/executeautomation/mcp-database-server
- MCP SQLite 参考实现集合: https://github.com/modelcontextprotocol/servers

注意：MCP 数据库类工具必须加权限边界，写操作需要后端二次确认；生产逻辑仍建议以 ORM Service 为准。

### 6.2 本地代码查看工具

本地代码查看需要兼顾“能看懂结构”和“安全可控”。推荐三层能力：

| 能力 | 推荐工具 | 用途 |
| --- | --- | --- |
| 语义级代码理解 | Serena | 查找符号、引用、文件结构，适合 Agent 理解本地项目 |
| 受控文件读取 | MCP Filesystem Server | 限定目录白名单，只允许读取用户授权代码库 |
| 快速文本搜索 | ripgrep | 本地快速搜索函数名、错误信息、配置项 |
| 代码库地图参考 | Aider Repo Map | 可借鉴其 repo map 思路，把大型代码库压缩为结构化上下文 |

GitHub 地址：

- Serena: https://github.com/oraios/serena
- MCP Filesystem Server: https://github.com/modelcontextprotocol/servers
- Aider: https://github.com/Aider-AI/aider
- ripgrep: https://github.com/BurntSushi/ripgrep
- Sourcegraph Cody: https://github.com/sourcegraph/cody

推荐落地方式：

- MVP 阶段先用 `ripgrep + 文件白名单读取 + Chroma 代码片段索引`。
- 增强阶段接入 Serena MCP，获得符号级检索、引用查找、代码结构概览。
- 专家模式接入 Codex CLI，专门处理复杂项目、宽技术栈、跨文件推理、架构级面试深挖。
- 不把 Codex CLI 作为默认代码查看链路，避免普通问题过重；默认模式仍保持轻量。

### 6.3 Codex CLI 专家模式

Codex CLI 作为“外部 Coding Agent Provider”接入，而不是替代系统内置 Agent。

适用场景：

- 项目目录很大，普通 RAG 片段检索难以理解整体架构。
- 技术栈跨度大，需要跨语言、跨目录、跨配置文件分析。
- 用户希望基于真实代码做面试项目深挖、技术亮点提炼、架构追问模拟。
- 需要更接近真实 Coding Agent 的代码库理解能力，但仍保持用户审阅权。

不适用场景：

- 普通文件搜索、函数解释、简历 / JD 匹配，这些仍走轻量链路。
- 默认自动修改代码。专家模式默认只读，不进入文件写入流程。

后端调用方式建议使用 `codex exec` 非交互模式：

```text
codex.cmd exec
  --cd <repo_path>
  --sandbox read-only
  --ask-for-approval never
  --json
  <prompt>
```

Windows 下优先调用 `codex.cmd`，避免 PowerShell 执行策略拦截 `codex.ps1`。

专家模式运行原则：

- 默认 `--sandbox read-only`，只允许阅读项目，不允许改文件。
- 默认 `--ask-for-approval never`，避免后端任务卡在交互确认。
- 禁止暴露 `--dangerously-bypass-approvals-and-sandbox` 给普通用户。
- 输出使用 `--json`，后端解析 JSONL 后转成系统 SSE 事件。
- 每次调用必须绑定 `space_id`、`session_id`、`repo_path`，避免多岗位上下文串线。

推荐事件映射：

```text
codex exec --json stdout
  -> event: codex_event
  -> event: source
  -> event: review
  -> event: codex_done
```

建议新增表：

```text
external_agent_runs
- id
- space_id
- session_id
- provider          # codex_cli / claude_code_future
- repo_path
- command
- status            # queued / running / completed / failed
- output_path
- error_message
- created_at
- updated_at
```

首次使用检测接口：

```text
GET /api/integrations/codex/status
```

检测逻辑：

```text
Windows: where codex.cmd
macOS / Linux: command -v codex
```

如果已安装，返回命令路径与版本；如果未安装，前端展示安装引导：

```text
1. 安装 Node.js
2. 运行 npm install -g @openai/codex
3. 运行 codex login
4. 回到系统点击“重新检测”
```

如果未来支持 Claude Code，可沿用同一个 `External Coding Agent Provider` 抽象，但 MVP 先只实现 Codex CLI。

### 6.4 网页查阅与溯源

理念中提到“能查阅网页，最好能让用户溯源”。当前阶段不购买外部搜索 API，因此先保留接口空挡，不把网页查阅作为 MVP 强依赖。

推荐抽象：

```text
WebResearchProvider
- OpenAIWebSearchProvider      # 预留
- ClaudeWebSearchProvider      # 预留
- GeminiGroundingProvider      # 预留
- DeepSeekSearchProvider       # 预留，等待官方 API 能力明确
- LocalCrawlerProvider         # 备用
```

当前落地策略：

- MVP 阶段先不接入付费网页搜索 API。
- 后端保留 `web_research` 工具接口、`sources` 表和 SSE 事件类型。
- Agent Prompt 中可以显式说明“当前未启用联网搜索”，避免模型假装查过网页。
- 用户手动粘贴 URL 或网页正文时，系统可以作为普通材料写入当前 Space。
- 后续有 API 预算后，优先接入 OpenAI Web Search、Claude Web Search 或 Gemini Grounding。

未来接入模型内置网页搜索时，后端统一返回：

```json
{
  "answer": "...",
  "sources": [
    {
      "title": "...",
      "url": "...",
      "snippet": "...",
      "provider": "openai"
    }
  ],
  "provider": "openai",
  "raw": {}
}
```

SSE 事件预留：

```text
event: web_search_start
event: web_source
event: token
event: web_search_done
```

建议新增表：

```text
sources
- id
- space_id
- session_id
- provider          # openai / claude / gemini / deepseek / local / manual
- title
- url
- snippet
- retrieved_at
- raw_json
```

如果后续必须本地抓取网页，再启用成熟开源方案：

- 搜索接口：可选 Tavily、Brave Search API 或 SerpAPI。
- 网页正文抽取：trafilatura / readability-lxml。
- 溯源存储：把 URL、标题、发布时间、抓取时间、引用片段写入 `tool_calls` 或 `sources` 表。
- 前端展示：在导师建议旁展示“参考来源”，用户可点击打开原网页。

本地抓取备用项目：

- Crawl4AI: https://github.com/unclecode/crawl4ai
- Firecrawl: https://github.com/firecrawl/firecrawl
- trafilatura: https://github.com/adbar/trafilatura
- browser-use: https://github.com/browser-use/browser-use
- Playwright: https://github.com/microsoft/playwright

## 7. 前端页面模块

### 7.1 Job Space 切换器

- 顶部或侧边栏展示当前 Space。
- 不同 Space 使用不同主题色：AI 工程师深海蓝、PM 钛金灰、日语学习樱花粉。
- 切换 Space 时，前端必须重新加载 `space_id` 下的 Resume、JD、Session、向量索引状态。

### 7.2 技术面试 Agent 页面

- 左侧：问题列表 / Session 历史。
- 中间：用户原始回答。
- 右侧或浮层：导师批改建议。
- 底部：Spotlight 式微调输入框。
- 操作：采纳并更新、保留原样、基于新指令重新生成。

### 7.3 阅读理解工作台

- 主区域只展示文章正文。
- 用户点击句子后，右侧栏展示语法拆解、背景释义、例句。
- 只记录阅读完成度、难句、用户标记，不引入复杂考试系统。

## 8. 安全与隔离

### 8.1 文件安全

- 本地代码读取必须由用户显式选择目录。
- 后端保存 allowlist，只允许访问授权路径。
- 禁止 Agent 读取系统目录、SSH Key、浏览器数据、环境变量文件等敏感路径。
- 文件工具默认只读，写文件能力不进入 MVP。

### 8.2 数据安全

- Resume、JD、面试回答都按 Space 隔离。
- Agent Tool 调用必须记录 `tool_calls`，便于审计。
- LLM 输出视为不可信输入，写库前必须经过后端 schema 校验。
- 数据库写操作必须走服务端函数，不允许前端或 Agent 直接提交任意 SQL。

### 8.3 Session 隔离

- 每次请求必须携带 `space_id` 与 `session_id`。
- 后端校验 `session.space_id == request.space_id`。
- SSE 连接只推送当前 Session 的事件。
- 短轮询状态接口只返回当前 Space / Session 范围内的数据。

## 9. MVP 实施顺序

### Phase 1：基础壳与数据模型

- FastAPI + SQLite + SQLAlchemy + Alembic。
- React + Vite + Tailwind + shadcn/ui。
- Job Space、Resume 上传、JD 上传、Session 创建。
- 短轮询任务状态接口。

### Phase 2：Agent 与 SSE

- LangChain 接入模型。
- LangGraph 实现面试批改流程。
- SSE 推送 token、tool、source、review 事件。
- 前端实现原回答 vs 导师建议对比视图。

### Phase 3：RAG 与本地代码读取

- Chroma 按 Space 建 collection。
- Resume / JD / 文章切片入库。
- MVP 先用 ripgrep + 白名单文件读取。
- 后续接入 Serena MCP 做语义级代码检索。

### Phase 3.5：Codex CLI 专家模式

- 增加 Codex 安装检测接口。
- 增加 `external_agent_runs` 表。
- 后端封装 `CodexExpertAdapter`。
- 使用 `codex exec --json --sandbox read-only` 运行只读分析。

## 10. 不推荐方案

| 方案 | 不推荐原因 |
| --- | --- |
| WebSocket 全量通信 | 本项目主要是单用户本地场景，SSE + 短轮询已经足够，WebSocket 增加连接维护成本 |
| 一开始使用 PostgreSQL | 本地运行复杂度上升，违背轻量目标；可作为后续多人部署选项 |
| Agent 直接写数据库 | 容易出现 SQL 注入、误删、越权写入；应通过后端 Service 与 ORM 控制 |
| 不做 Space 隔离 | 不同岗位 Resume / JD / 代码上下文会串线，直接破坏“分岗管理”理念 |
| 自研代码语义检索 | 成本高且容易不稳定，优先使用 Serena、MCP Filesystem、ripgrep 等成熟方案 |
| Codex CLI 作为默认检索链路 | 普通问题会变慢变重；应作为专家模式，仅在复杂项目分析时启用 |
| MVP 强依赖付费网页搜索 API | 当前预算不支持，先保留 Provider 空挡和手动材料导入能力 |

## 11. 最终推荐

本项目最优路线是：

```text
React + Vite 前端
  -> HTTP REST / 短轮询 / SSE
  -> FastAPI 后端
  -> LangChain + LangGraph Agent
  -> SQLite + SQLAlchemy + Alembic
  -> Chroma 本地向量库
  -> Serena / MCP Filesystem / ripgrep 本地代码工具
  -> Codex CLI 专家模式
  -> WebResearchProvider 预留网页搜索能力
```

这个组合能同时满足：

- 前后端短轮询要求。
- SSE 流式 Agent 输出。
- 基于 HTTP，无需复杂握手和协议维护。
- 本地轻量运行。
- 多岗位、多 Resume、多 JD、多 Session 强隔离。
- Agent 工具化能力。
- 成熟开源工具支撑数据库操作与本地代码查看。
- Codex CLI 支撑复杂项目的专家级代码理解。
- 网页搜索能力保留扩展口，当前不强依赖付费 API。
- 保留后续迁移 PostgreSQL、多用户部署、更多 Agent 工具的扩展空间。
